调试前

需要用g++ -g选项编译，使生成满足GDB要求的可执行文件

也可以用-O0~-O4和-g一起，表示优化，更友好的是-Og

调试

  • 普通调试
    • 启动调试
      • gdb + [参数] 文件名 直接链接程序

        gdb 文件名 → l列出各行 → 设置断点 → r 或 start → c或 n → p → q

        --args 用于向程序传递参数，如 gdb --args main astring

      • gdb 启动调试

        gdb开启之后，可以用 file 文件名 命令定位要调试的程序，再 run或者直接用 run 文件名 启动目标程序用 set args astring 传递参数

        用 cd 改变gdb调试器的工作目录

        用 path 临时修改环境变量

    • 输入输出重定向

      如 run > a.txt 表示将程序的执行结果输出到a.txt文件中

    • 断点
      • 普通断点
        • break两种方式：
          (gdb) break location
(gdb) break ... if cond #b 7 if num>10

          其中，location/... 有多种表示方式，如下：

        • tbreak 和 rbreak

          tbreak: tbreak 命令打的断点仅会作用 1 次，即使程序暂停之后，该断点就会自动消失

          rbreak: rbreak 命令的作用对象是 C、C++ 程序中的函数，它会在指定函数的开头位置打断点

          (gdb) tbreak location
(gdb) tbreak location
(gdb) rbreak regex #regex是正则表达式，如rbreak rb_*
      • 观察断点

        软件观察点：以单步执行的方式运行程序，并且每行代码执行完毕后，都会检测该目标变量（表达式）的值是否发生改变，如果改变则程序执行停止

        硬件观察点：系统会为 GDB 调试器提供少量的寄存器（例如 32 位的 Intel x86 处理器提供有 4 个调试寄存器），每个寄存器都可以作为一个观察点协助 GDB 完成监控任务

        (gdb) watch var #只有当被监控变量（表达式）的值发生改变，程序才会停止运行
(gdb) rwatch var #只要程序中出现读取目标变量（表达式）的值的操作，程序就会停止运行
(gdb) awatch var #只要程序中出现读取目标变量（表达式）的值或者改变值的操作，程序就会停止运行
(gdb) info watchpoints #查看当前建立的观察点的数量
      • 捕捉断点
        (gdb) catch event
(gdb) up #回到异常发生的源码处
      • 添加条件
        • condition
          (gdb) condition bnum expression #用于为某断点添加条件，bnum是断点编号
(gdb) condition bnum #用于删除某断点的条件

        • ignore
          (gdb) ignore bnum count #count 表示某断点失效的次数
    • 单步调试

      next ：调用的函数，next 命令只会将其视作一行代码

      step ：当 step 命令所执行的代码行中包含函数时，会进入该函数内部，并在函数第一行代码处停止执行

      until ：快速执行完循环体（只有在函数体内最后一步才生效），也可以加行号，表示执行至指定位置停止

  • 调试正在运行的程序

    pidof 文件名查出PID → gdb -p PID 连接到程序上 → 调试

  • 调试异常崩溃的程序

    在 Linux 操作系统中，当程序执行发生异常崩溃时，系统可以将发生崩溃时的内存数据、调用堆栈情况等信息自动记录下载，并存储到一个文件中，该文件通常称为 core 文件，Linux 系统所具备的这种功能又称为核心转储（core dump）

    通过ulimit -c unlimited 命令开启core dump功能，程序崩溃会在当前目录下生成core文件，用gdb 文件名 core可以查看崩溃的位置
